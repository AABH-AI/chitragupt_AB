# Project Conventions & Engineering Protocols

**Phase:** System Architecture & Implementation Setup
**Version:** 1.0 — Binding Standard
**Target Stack:** Python 3.11+, LangGraph, FastAPI, pgvector (PostgreSQL 16+), Redis, AWS ECS Fargate, S3

---

## 1. Code Quality & Language Standards

To ensure a highly maintainable, type-safe, and self-documenting codebase, the Chitragupt engineering team must adhere to the following standards:

### 1.1 Python Language Standards

- **Formatting:** Code formatting must strictly adhere to the `black` code style with a line length of 88 characters.
- **Imports:** Imports must be sorted alphabetically using `isort` with the black-compatible configuration.
- **Type Hinting:** Mandatory static type hints on all function signatures, variable assignments, and class definitions. Type-checking must pass `mypy` under strict mode.
- **Documentation:** Every class, public method, module, and API route handler must contain a triple-quoted Google-style docstring outlining arguments, return types, and side effects.

### 1.2 LangGraph Node & Stateful Design

- **State Immutability:** The LangGraph `State` class must use immutable `Pydantic` models or standard Python typing `TypedDict` with deep-copied dictionary states to prevent hidden mutations across parallel paths.
- **Node Determinism:** Every node within the LangGraph orchestrator must represent a deterministic single-step transaction (e.g., retrieving chunks, applying a prompt, parsing JSON). Complex multi-step reasoning must be split into separate, observable nodes.
- **State Appenders:** Accumulator lists (e.g., listing gaps or requirements) must use LangGraph's native `annotated` list appender pattern rather than manual assignment, ensuring concurrency-safe updates.

## 2. Git Branching & Commit Protocol

We enforce a strict branching and Conventional Commits standard to ensure traceability of features directly to our Sprint goals.

### 2.1 Branch Naming Conventions

All branch names must start with a scoped prefix followed by the ticket ID or short description:

- `feat/sprint-[X]-[short-name]` — For new functional components matching Sprint deliverables.
- `fix/[bug-id]-[short-name]` — For bug fixes and regression patches.
- `chore/[short-name]` — For CI/CD updates, dependency upgrades, or documentation changes.
- `perf/[short-name]` — For latency, semantic search, or query optimization updates.

### 2.2 Conventional Commits Specification

Commit messages must strictly follow the Conventional Commits format to facilitate automated changelog generation:

```text
<type>(<scope>): <description>

[optional body]

[optional footer(s)]
```

- **Allowed Types:** `feat` (new feature), `fix` (bug fix), `docs` (documentation changes), `style` (formatting, missing semi-colons), `refactor` (code restructuring with no behavior change), `perf` (performance improvement), `test` (adding missing tests), `chore` (maintenance task).
- **Scope Examples:** `rag`, `agent`, `db`, `auth`, `jira`, `export`, `ui`.
- **Breaking Changes:** Must carry a `BREAKING CHANGE:` footer or an exclamation mark after the type/scope (e.g., `feat(db)!: migration to pgvector schema`).

## 3. Database & pgvector Integration Protocols

To protect multi-tenancy and guarantee sub-second semantic search latency, all SQL and vector operations must follow these rules:

### 3.1 Strict Tenant Isolation (RLS)

- **PostgreSQL Row-Level Security:** In strict compliance with **INV-SEC-01**, every single table containing tenant-specific data (including `Document`, `Chunk`, `Requirement`, `Conflict`, `Gap`) must have PostgreSQL Row-Level Security enabled.
- **Policy Enforcement:** All application queries must set the active transaction session variable `app.current_tenant_id` at the start of a transaction block. Every table must apply the following policy:
  
  ```sql
  ALTER TABLE chunks ENABLE ROW LEVEL SECURITY;
  CREATE POLICY tenant_isolation_policy ON chunks
    USING (tenant_id = NULLIF(current_setting('app.current_tenant_id', true), '')::UUID);
  ```

- **ORMs:** The use of raw database sessions is restricted. The database service wrapper must automatically attach the current tenant's context to every transaction block, preventing developers from manually omitting the filter.

### 3.2 Vector Storage & Similarity Queries

- **Consistency:** The active vector schema must consistently represent 1536-dimensional embeddings generated by `text-embedding-3-large` or `voyage-large-2` (**INV-MODEL-05**).
- **Query Bounds:** All vector similarity lookups must include:
  1. An active tenant filter (automatically handled by database RLS).
  2. A `similarity_threshold` parameter (lookups returning similarity scores below 0.65 must be dropped to prevent semantic noise).
  3. A strict `limit` (K value) capped at a maximum of 15 context chunks per query.
- **Index Optimization:** Once a vector table exceeds 50,000 active rows, System Architects must provision a Hierarchical Navigable Small World (HNSW) index using Cosine Distance (`vector_cosine_ops`) to guarantee sub-second search latencies under concurrent load.

## 4. Upstream LLM & Fallback Protocols

Our systems must remain resilient to upstream outages and rate limits while maintaining optimal quality.

### 4.1pinned Models

Production code must never reference floating or dynamic model aliases. Every LLM invocation must use pinned provider identifiers (**INV-MODEL-03**):

- Primary Quality/Reasoning Engine: `claude-3-5-sonnet-20241022`
- Fast/Classification Engine: `claude-3-5-haiku-20241022`
- Fallback/Large-Context Engine: `gemini-1.5-pro-002`

### 4.2 API Resilience & Fallback Routing

- **Retry Policy:** Every LLM API client call must be wrapped in a retry handler with exponential backoff and jitter (maximum 3 retries).
- **Fallback Chain:** If a primary call to Anthropic Claude fails with a 5xx error or a 429 Rate Limit exception after retries, the model client must instantly catch the exception and route the execution context to Google Gemini 1.5 Pro using the same LangGraph state structure. This fallback execution must raise a system flag so that the event is logged in our Langfuse trace suite.
- **Zero Retention Policy:** Every LLM client wrapper must be explicitly configured to connect to enterprise-tier endpoints that guarantee a zero data retention policy for model training.

## 5. Traceability & Epistemological Protocols

To prevent AI hallucination and build absolute user trust, all generated specifications must be rooted in strict factuality:

### 5.1 Requirement Grounding

- **Source Coordinates:** A generated requirement is considered invalid if it does not contain a populated `source_chunks` metadata array (**INV-EPI-01**).
- **Granular Linkage:** Every chunk reference must track the `document_id`, `chunk_id`, file name, and exact line/paragraph coordinates.
- **Inference Tagging:** Any requirement generated through cross-document semantic synthesis or conceptual deduction must carry the `[INFERRED — REVIEW REQUIRED]` flag. Any requirement drawn from an visual diagram extraction must carry the `[VISUAL EXTRACTION — VERIFY]` flag with its confidence score capped at a maximum of 0.80.

### 5.2 Conflict & Gap Tracking

- **Conflict Escalation:** The system must never attempt to resolve contradictions automatically. If the search agent retrieves contradicting source chunks of equal trust tiers, it must flag the affected items with `[CONFLICT DETECTED — PENDING]` and immediately halt the requirement's state progress.
- **Human Resolution:** A conflict is resolved strictly through human intervention via the side-by-side UI panel. The BA's selection is written as absolute ground truth and locked (`INV-HITL-01`).

## 6. Budget & Cost-Containment Circuit Breakers

To prevent rogue agent loops from exhausting organization API limits or exceeding financial tolerances, the system enforces strict cost checkpoints:

### 6.1 Token Caching Protocols

- **System Prompt Caching:** LangGraph agents must implement prompt caching for all static system prompt headers and schema definitions, cutting active context token ingestion costs by up to 90%.
- **Source Document Caching:** Large document inputs in a project context must be cached at the LLM provider layer if they are queried sequentially within the same elicitation chat session.

### 6.2 Cost Circuit Breakers

- **Project Budget Cap:** In accordance with **INV-COST-01**, every active requirements project is assigned a hard budget limit (default: $20.00). The agent execution engine must query the `llm_call_logs` table before spawning any node. If the current accumulated project cost reaches 80% of the cap, a priority email alert is dispatched to the project owner. If cost reaches 100%, all agent executions for that project are instantly suspended.
- **Workspace Budget Cap:** In accordance with **INV-COST-02**, every workspace has a monthly cap. If exceeded, all execution threads across all projects inside the workspace are paused until the cap is raised.

## 7. Test-Driven Invariant Verification

To ensure that system invariants are never broken during codebase changes or prompt refactoring, developers must implement the following automated test assertions in the CI/CD pipeline:

### 7.1 Automated Tenant Isolation Test (Verification of INV-SEC-01)

- Every backend unit test suite must run a cross-tenant simulation:
  1. Initialize Tenant A and Tenant B.
  2. Index documents for Tenant A.
  3. Attempt to execute a semantic lookup query under the context of Tenant B.
  4. Assert that the returned result list is strictly empty (`len(results) == 0`).
  5. Attempt a raw SQL query on `chunks` under Tenant B's session and assert that 0 rows are returned.

### 7.2 Automated Grounding Verification (Verification of INV-EPI-01)

- Write a regression integration test that feeds a set of requirements inputs to the synthesis agent:
  1. Capture the output JSON payload.
  2. Parse the requirements array.
  3. Assert that every item in the list carries a valid `source_chunks` key pointing to active vector primary keys present in the database.
  4. Assert that any requirement without a source citation fails schema validation and is automatically rejected from the database commit queue.

### 7.3 Automated Human Override Verification (Verification of INV-HITL-01)

- Validate our review-loop integrity:
  1. Generate requirement draft `REQ-001`.
  2. Simulate a human review action modifying the text: `REQ-001.description = "Custom Human Override Text"` and status = `BA Approved`.
  3. Re-run the synthesis pipeline on the same source documents.
  4. Assert that the database row for `REQ-001.description` is completely unchanged and preserves the human override text.
  5. Assert that the synthesis agent's suggestions are saved to an isolated version history table rather than overwriting the approved active record.
