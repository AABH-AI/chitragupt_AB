# Epistemology: Theory of Knowledge for Chitragupt

**Phase:** Product Discovery
**Version:** 2.0 — Expanded
**Purpose:** To define how the Business Requirement Analyzer (Chitragupt) acquires, validates, represents, and evolves knowledge. In a RAG-based agentic system, epistemology is not academic — it directly governs which outputs the system is allowed to produce, how they are tagged, and what the human reviewer must verify.

---

## 1. The Hierarchy of Truth

The system must evaluate conflicting or overlapping information based on a strict hierarchy of trust. Every claim in the generated specification must be traceable to one of these tiers.

| Rank | Source | Trust Level | Examples |
| :---: | :--- | :--- | :--- |
| 1 | **Human Override** | Absolute | Reviewer's inline edits, explicit approvals, rejections |
| 2 | **Structured External Systems** | High | Jira custom fields, database schemas, typed API contracts |
| 3 | **Primary Source Documents** | Baseline | Client PDFs, official Confluence pages, signed emails |
| 4 | **Secondary Documents** | Moderate | Meeting notes, Slack threads, internal memos |
| 5 | **Agentic Inference** | Conditional | LLM synthesis of implicit intent; industry standard assumptions |

**Critical rule:** A claim derived from Rank 5 (Agentic Inference) can never supersede or silently replace a claim from Rank 3 or above. If conflict exists between tiers, the higher-ranked source wins and the lower-ranked claim is discarded or flagged — never merged silently.

## 2. Knowledge Provenance (Traceability)

For a requirement to be considered "known," the system must be able to prove *how* it knows it. Provenance is not optional metadata — it is a first-class structural property of every knowledge object in the system.

### 2.1 Atomic Traceability

Every generated requirement, constraint, or assumption must maintain:

- **Source ID:** The document or session from which the evidence was retrieved.
- **Chunk ID:** The exact vector chunk from the source.
- **Retrieval Score:** The cosine similarity (or re-ranking score) between the query and the chunk at retrieval time.
- **Agent ID:** Which agent in the pipeline generated or synthesized this requirement.
- **Model ID:** Which LLM model and version produced the output.
- **Timestamp:** When the synthesis occurred.
- **Confidence Score:** A float (0.0–1.0) representing the system's certainty.

### 2.2 Citation Requirement

The output specification must include visible, human-readable citations for every non-trivial claim:

```
[Source: Payment_Gateway_V2.pdf, Page 4, Paragraph 2 — Confidence: 0.92]
[Source: Jira Epic ENG-1234 — Confidence: 0.87]
[INFERRED from industry standard (GDPR Art. 17) — Confidence: 0.61 — REQUIRES VALIDATION]
```

### 2.3 Orphan Knowledge

Any claim generated by the LLM that cannot be mapped back to at least one chunk in the vector store is defined as **Orphan Knowledge**. Orphan Knowledge is treated as a hallucination until proven otherwise:

- The system must never include Orphan Knowledge in the final specification without an explicit `[INFERRED — NO SOURCE]` tag.
- The Review & Feedback Agent must actively prune Orphan Knowledge and surface it as an Open Question rather than including it as a requirement.

## 3. Degrees of Certainty & Confidence

The system does not treat all generated text equally. Every synthesized output must be assigned a confidence tier based on how it was derived.

| Confidence Tier | Label | Score Range | Description | User Tag |
| :--- | :--- | :--- | :--- | :--- |
| **Explicit Extraction** | High | 0.85–1.00 | Direct quote or verbatim restatement of source text | *(no tag)* |
| **Deductive Synthesis** | Medium | 0.65–0.84 | Logical combination of two or more explicit facts | `[SYNTHESIZED]` |
| **Inductive Inference** | Low | 0.40–0.64 | Pattern-based inference using industry norms | `[INFERRED — VERIFY]` |
| **Speculative Extension** | Very Low | < 0.40 | Extrapolation with insufficient evidence | `[SPECULATIVE — REVIEW]` |

**Operational thresholds:**
- Scores < 0.40: The system must NOT include the claim in the spec. It must raise it as an Open Question instead.
- Scores 0.40–0.64: Include with `[INFERRED — VERIFY]` tag; flag in the Open Questions list.
- Scores 0.65–0.84: Include with `[SYNTHESIZED]` tag; reviewable but acceptable as draft.
- Scores ≥ 0.85: Include without tag; considered high-confidence grounded output.

### 3.1 Confidence Calibration

The system's confidence scores must be *calibrated* — a score of 0.70 should correspond to approximately 70% empirical correctness when evaluated against human expert review. Calibration must be validated:

1. **At launch:** Use the evaluation dataset (5–10 historical requirement sets) to benchmark predicted vs actual confidence.
2. **Ongoing:** Each human approval/rejection feeds a calibration signal back into the confidence model.
3. **Recalibration trigger:** If the human rejection rate for claims scored > 0.85 exceeds 15%, the confidence model is considered miscalibrated and must be reviewed.

## 4. Handling Contradictions

When two sources of equal hierarchy conflict, the system faces an epistemological impasse. It must not attempt to resolve the contradiction autonomously.

### 4.1 Conflict Classification

| Conflict Type | Example | System Action |
| :--- | :--- | :--- |
| **Direct Contradiction** | Source A: "Daily sync required"; Source B: "Real-time sync required" | Raise Conflict Flag; halt synthesis for that topic |
| **Scope Overlap** | Source A specifies feature X for Admin; Source B specifies the same for All Users | Raise ambiguity; synthesize both interpretations as candidates |
| **Version Conflict** | Same document exists in v1 (uploaded) and v2 (linked URL); they differ | Flag version discrepancy; use most recent by default; notify user |
| **Implicit Contradiction** | Feature requirement implies SLA X; constraint document implies max SLA Y (where X > Y) | Raise inferred conflict; tag both; ask stakeholder to clarify |

### 4.2 Conflict Resolution Protocol

The system does **not** possess the epistemological authority to guess which conflicting source is correct. The protocol is:

1. Both conflicting sources are surfaced as a **Conflict Object** with full citations.
2. Synthesis for the affected topic is paused (partially completed sections are marked `[CONFLICT — PENDING RESOLUTION]`).
3. The Conflict Object is surfaced in the UI with a side-by-side comparison of the two sources.
4. A human stakeholder resolves the conflict via one of three actions:
   - **Accept Source A** — marks Source A as ground truth for this topic.
   - **Accept Source B** — marks Source B as ground truth for this topic.
   - **Provide Clarification** — human enters a new authoritative statement (becomes Rank 1 truth).
5. Once resolved, synthesis resumes using the accepted or clarified statement.

## 5. Temporal Knowledge & Document Versioning

Knowledge in the BRA system has a **temporal dimension**: a fact that was true when a document was ingested may become false when that document is updated.

### 5.1 Knowledge Validity Window

Every embedded chunk must carry a **valid_from** timestamp (the time the source was ingested) and optionally a **valid_until** timestamp (set when the source document is updated or superseded).

- Chunks with expired `valid_until` must be **excluded from retrieval** (filtered at query time by the vector store metadata filter).
- When a document is re-ingested after an update, the old chunks are tombstoned (marked inactive) rather than deleted — preserving the audit trail.

### 5.2 Document Version Management

| Event | System Response |
| :--- | :--- |
| Source document updated (new version uploaded) | Old chunks tombstoned; new version re-chunked and re-embedded; spec flagged for re-review |
| Jira epic updated after ingestion | Re-ingestion triggered by webhook; diff-based re-embedding (only changed sections) |
| User manually marks a source as "superseded" | All chunks from that source excluded from future retrieval; existing requirements linked to it flagged |
| Spec is locked and approved | All contributing source chunks are snapshotted; changes to source docs after lock do not affect the approved spec |

### 5.3 Knowledge Decay & Staleness Detection

The system must proactively detect and surface knowledge that may have decayed:

- If a source document has not been verified by a human in > 90 days, requirements derived from it are marked `[SOURCE MAY BE STALE — VERIFY]`.
- If a Jira epic or Confluence page has been modified since ingestion (detected via API polling or webhook), a **staleness alert** is raised in the project dashboard.
- The staleness threshold (90 days default) must be configurable per organization.

## 6. Multi-Modal Knowledge Epistemology

The BRA system accepts inputs beyond plain text — images, diagrams, voice recordings, and spreadsheets each contribute knowledge differently and must be treated accordingly.

### 6.1 Knowledge Contribution by Modality

| Input Modality | Knowledge Type | Extraction Method | Trust Level | Special Consideration |
| :--- | :--- | :--- | :--- | :--- |
| **Plain text (PDF/DOCX)** | Explicit statements | Direct text extraction + chunking | Baseline | Tables and structured data require special parsing |
| **Spreadsheet (XLSX)** | Structured/tabular requirements | Schema-aware parser; row = potential requirement | High (structured) | Column headers define the semantic schema |
| **Architecture diagrams / images** | System entities, relationships, implicit constraints | Vision LLM (GPT-4o Vision / Claude vision) | Moderate (vision inference) | Vision extractions must be tagged `[VISUAL EXTRACTION — VERIFY]` |
| **Audio recordings** | Stakeholder intent, verbal agreements, exceptions | Whisper transcription → text pipeline | Moderate | Speaker attribution affects trust level |
| **Video (screen recording)** | UI flows, system behavior demonstrations | Frame extraction + transcription | Moderate | Treat as secondary confirmation, not primary source |
| **URL / Web page** | Context, public-facing requirements | Web scraper + HTML parser | Baseline | Mark as `source_type: web`; cannot verify authoritativeness |
| **Chat session (elicitation)** | Real-time stakeholder clarification | Direct LLM input; already in text form | High (recency) | Chat is authoritative for the question asked but may contradict older docs |

### 6.2 Vision Extraction Protocol

When a diagram or image is the input:

1. The vision-capable LLM (GPT-4o with vision or Claude with image support) is invoked to describe the diagram's entities and relationships.
2. The extracted description is converted to text and embedded as a chunk with `source_type: visual_extraction` and `confidence_modifier: -0.15` (visual extractions are inherently less precise than direct text).
3. All vision-extracted claims are tagged `[VISUAL EXTRACTION — VERIFY]` in the final spec regardless of confidence score.
4. The original image is stored and linkable from the citation viewer so the reviewer can verify the extraction manually.

## 7. Confidence Calibration Methodology

Confidence calibration transforms a raw LLM probability or heuristic score into a meaningful, empirically validated certainty estimate.

### 7.1 Calibration Inputs

| Signal | Weight | Description |
| :--- | :--- | :--- |
| **Retrieval score** | 40% | Cosine similarity between query and retrieved chunk. High = better match. |
| **Re-ranking score** | 30% | Cross-encoder re-ranking score after initial retrieval. More precise than cosine. |
| **Source hierarchy tier** | 15% | Higher-tier sources (Rank 2–3) receive a confidence boost. |
| **LLM self-reported certainty** | 10% | LLM asked to rate its own confidence via structured output. Known to be overconfident; discounted. |
| **Historical validation rate** | 5% | Empirical rate at which similar past inferences were approved by human reviewers. |

### 7.2 Calibration Evaluation Protocol

The product team must establish a calibration evaluation protocol before launch:

1. Sample 100 requirements from the evaluation dataset.
2. Have 2 domain experts independently rate each as: Correct / Incorrect / Partially Correct.
3. Plot predicted confidence scores vs. empirical accuracy (calibration curve).
4. Target: Brier Score < 0.15 (well-calibrated); Expected Calibration Error (ECE) < 0.10.
5. Re-evaluate calibration quarterly as the LLM models are updated.

## 8. Knowledge Evolution & Feedback Loops

The BRA system must improve its knowledge quality over time based on human reviewer behavior.

### 8.1 Feedback Signals & Their Meaning

| Human Action | Signal Captured | Effect on System |
| :--- | :--- | :--- |
| Approves a requirement | Positive signal for: retrieval query, chunks used, synthesis approach | Used in calibration; boosts similar chunks' rank in future retrievals for this domain |
| Rejects a requirement | Negative signal; captures what was wrong (via optional rejection reason) | Chunk receives a soft negative weight for this retrieval context |
| Edits a requirement | Delta between generated and human version is logged | Informs prompt template refinement; identifies systematic phrasing patterns |
| Resolves a Conflict (chooses A or B) | Conflict resolution pattern recorded | Similar conflicts in same domain use resolved pattern as a hint (soft, not override) |
| Marks an `[INFERRED]` as valid | Validates the inference pattern | Raises confidence floor for similar inductive inferences in this domain |
| Marks an `[INFERRED]` as invalid | Invalidates the inference | Lowers confidence for similar inferences; triggers review of related claims |

### 8.2 Organizational Knowledge Accumulation

Beyond per-project feedback, the system must accumulate organizational knowledge:

- **Approved specification library:** Every locked and approved specification becomes a reference corpus for future projects in the same domain.
- **Domain glossary enrichment:** Terms defined or clarified by human reviewers are added to the domain glossary and surfaced during future elicitation.
- **Template refinement:** Recurring gap types (e.g., missing SLA requirements on every fintech project) are added to the domain-specific completeness checklist automatically.
- **Conflict pattern library:** Resolved conflicts are logged with their resolution rationale; similar conflicts are pre-flagged in future projects.

### 8.3 Privacy & Isolation of Feedback

Organizational learning must respect tenant isolation:

- Feedback from Tenant A never directly modifies the knowledge base used by Tenant B.
- Cross-tenant learning is permitted **only** for: aggregate calibration statistics (no identifiable data), anonymized conflict resolution patterns contributed to a shared library with explicit opt-in.

## 9. Knowledge Boundaries & Scope Limitation

The system must know what it does not know, and it must say so explicitly rather than speculate.

### 9.1 Out-of-Knowledge Scenarios

The following scenarios must result in an Open Question, not a synthesized requirement:

| Scenario | Correct System Response |
| :--- | :--- |
| No source chunk covers the topic at all | Raise Open Question: "No information found for [topic]. Please provide clarification." |
| All retrieved chunks have confidence < 0.40 | Raise Open Question; do not include speculative claim |
| Topic requires domain expertise beyond the ingested corpus | Raise Open Question: "This topic may require subject matter expert input." |
| Topic is explicitly marked out-of-scope in a source document | Do not generate requirement; note the exclusion with citation |
| Stakeholder input contradicts itself within a single session | Raise internal conflict; ask for clarification before proceeding |

### 9.2 Epistemic Humility in Output Language

The system's language model must be instructed to use appropriately hedged language that reflects uncertainty:

- **High confidence:** "The system shall..." (mandatory language)
- **Medium confidence:** "Based on [source], the system should..." (recommended)
- **Low confidence:** "It is recommended that the system considers..." (tentative; tagged `[INFERRED]`)
- **No basis:** Do not generate. Raise as Open Question.

The system must never use certain language ("shall," "must") for inferred claims. This is a hallucination risk and a liability risk for the organization using the spec.
