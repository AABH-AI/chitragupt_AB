# Epistemology: Theory of Knowledge for Chitragupt

**Phase:** Product Discovery
**Purpose:** To define how the Business Requirement Analyzer (Chitragupt) acquires, validates, and trusts knowledge. In a RAG-based agentic system, epistemology determines the boundaries between factual retrieval, inferred synthesis, and potential hallucination.

---

## 1. The Hierarchy of Truth

The system must evaluate conflicting or overlapping information based on a strict hierarchy of trust.

1. **Human Override (Absolute Truth):** Any edit, approval, or explicit rule set by a human reviewer supersedes all other knowledge.
2. **Structured External Systems (High Trust):** Data ingested from strictly typed systems (e.g., Jira custom fields, Database schemas) is trusted over unstructured text.
3. **Primary Source Documents (Baseline Truth):** Ingested PDFs, Confluence pages, and client emails.
4. **Agentic Inference (Conditional Truth):** Knowledge synthesized by the LLM that is not explicitly stated in a source document. This must *always* be flagged for human verification.

## 2. Knowledge Provenance (Traceability)

For a requirement to be considered "known," the system must be able to prove *how* it knows it.

- **Atomic Traceability:** Every generated requirement or constraint must maintain a direct cryptographic or database link to the source chunk(s) it was derived from.
- **Citation Requirement:** The output specification must include visible citations (e.g., `[Source: Payment_Gateway_V2.pdf, Pg 4]`).
- **Orphan Knowledge:** Any claim generated by the LLM that cannot be mapped back to the vector store is considered an "Orphan" (hallucination) and must be pruned or flagged.

## 3. Degrees of Certainty & Confidence

The system does not treat all generated text equally. The agentic layer must assign a confidence score to its outputs.

- **Explicit Extraction (High Confidence):** The LLM extracts a direct quote or explicitly stated rule (e.g., "The system must support 10,000 TPS").
- **Deductive Synthesis (Medium Confidence):** The LLM combines two explicit facts to form a logical conclusion.
- **Inductive Inference (Low Confidence):** The LLM infers a missing requirement based on industry standards (e.g., inferring GDPR compliance because the client is in Europe). These must be tagged with `[INFERRED - REQUIRES VALIDATION]`.

## 4. Handling Contradictions

When two sources of equal hierarchy conflict (e.g., a Jira ticket says "Daily Sync" and a PDF says "Real-time Sync"):

- The system does **not** possess the epistemological authority to guess which is correct.
- The contradiction itself becomes a first-class knowledge object (a "Conflict Flag") that is surfaced to the human.
